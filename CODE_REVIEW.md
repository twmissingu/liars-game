# 代码审查指南

本文档定义了 Liars Game 项目的代码审查规范，旨在确保代码质量、维护性和一致性。

---

## 📋 审查原则

### 核心原则

1. **建设性反馈** - 审查的目的是改进代码，而非批评人
2. **尊重与专业** - 保持礼貌，专注于代码本身
3. **及时响应** - 尽快完成审查，避免阻塞开发流程
4. **知识共享** - 审查是学习和分享知识的机会

### 审查者责任

- 确保代码符合项目规范
- 发现潜在 Bug 和安全问题
- 提供改进建议
- 帮助提升代码质量

### 作者责任

- 确保代码在提交前已自测
- 提供清晰的 PR 描述
- 积极响应审查意见
- 保持开放心态接受反馈

---

## ✅ 审查清单

### 1. 代码质量

#### 可读性
- [ ] 代码是否易于理解？
- [ ] 命名是否清晰、有意义？
- [ ] 函数是否足够简短（建议 < 50 行）？
- [ ] 是否有适当的注释说明复杂逻辑？

#### 一致性
- [ ] 代码风格是否符合项目规范？
- [ ] 是否使用了统一的命名约定？
- [ ] 缩进和格式是否一致？

#### 可维护性
- [ ] 代码是否遵循单一职责原则？
- [ ] 是否有重复代码需要提取？
- [ ] 依赖关系是否清晰？
- [ ] 是否有适当的错误处理？

### 2. 功能正确性

- [ ] 功能是否按需求实现？
- [ ] 边界条件是否处理？
- [ ] 异常情况是否有处理？
- [ ] 游戏逻辑是否正确？

### 3. 类型安全

- [ ] TypeScript 类型是否正确定义？
- [ ] 是否避免了 `any` 类型？
- [ ] 接口定义是否清晰？
- [ ] 返回值类型是否正确？

### 4. 测试

- [ ] 是否有足够的单元测试？
- [ ] 测试是否覆盖了主要逻辑？
- [ ] 测试用例是否清晰？
- [ ] 所有测试是否通过？

### 5. 性能

- [ ] 是否有明显的性能问题？
- [ ] 是否避免了不必要的重渲染？
- [ ] 资源加载是否优化？
- [ ] 内存使用是否合理？

### 6. 安全性

- [ ] 是否有 XSS 风险？
- [ ] 用户输入是否验证？
- [ ] 敏感信息是否泄露？
- [ ] 依赖包是否有已知漏洞？

### 7. 文档

- [ ] 公共 API 是否有 JSDoc 注释？
- [ ] 复杂逻辑是否有注释？
- [ ] README 是否需要更新？
- [ ] CHANGELOG 是否更新？

## 📝 审查评论规范

### 评论分类

| 标签 | 含义 | 示例 |
|-----|------|------|
| **🔴 [阻塞]** | 必须修复才能合并 | "这里会导致内存泄漏" |
| **🟡 [建议]** | 建议修改，但非必须 | "可以考虑使用 useMemo 优化" |
| **🟢 [疑问]** | 需要澄清的问题 | "这个逻辑在什么情况下会触发？" |
| **💡 [提示]** | 分享知识或技巧 | "这里可以用解构赋值简化" |
| **👍 [赞赏]** | 对好代码的肯定 | "这个实现很优雅！" |

### 有效评论示例

**✅ 好的评论**

```
🔴 [阻塞] 这里需要添加错误处理，如果 player 为 null 会导致崩溃。

建议：
```typescript
if (!player) {
  throw new Error('Player not found');
}
```
```

```
🟡 [建议] 这个函数可以拆分成更小的函数，提高可读性。

当前：
```typescript
function handleGameLogic() {
  // 50+ 行代码
}
```

建议：
```typescript
function handleGameLogic() {
  validateGameState();
  processPlayerTurn();
  checkWinCondition();
}
```
```

```
💡 [提示] 这里可以使用可选链操作符简化：

```typescript
// 当前
const name = player && player.name;

// 建议
const name = player?.name;
```
```

**❌ 避免这样评论**

```
这里不对。
```

```
改一下。
```

```
这个太烂了。
```

## 🔄 合并策略

### 合并要求

1. **至少 1 名审查者批准**（核心功能需要 2 名）
2. **所有 CI 检查通过**
3. **所有阻塞性问题已解决**
4. **PR 与目标分支无冲突**

### 合并方式

| 场景 | 合并方式 | 说明 |
|-----|---------|------|
| 功能分支 → develop | **Squash and Merge** | 保持 develop 历史整洁 |
| release → main | **Merge Commit** | 保留完整发布历史 |
| hotfix → main | **Merge Commit** | 保留紧急修复记录 |

### Squash Merge 规范

合并信息格式：

```
<type>(<scope>): <subject>

<详细描述>

Closes #<PR编号>
```

## 🎯 特殊场景处理

### 大型 PR

如果 PR 改动超过 500 行：

1. 建议拆分成多个小 PR
2. 如果无法拆分，安排线下代码走查
3. 增加审查时间预期

### 紧急修复

1. 可以简化审查流程
2. 但仍需至少 1 人快速审查
3. 事后补充测试和文档

### 重构 PR

1. 确保有充分的测试覆盖
2. 功能行为应保持一致
3. 分步骤审查，先逻辑后细节

## 📚 审查最佳实践

### 审查者

1. **先理解再评论** - 确保理解代码意图
2. **区分风格与问题** - 风格问题用建议，逻辑问题用阻塞
3. **提供解决方案** - 不仅指出问题，还提供改进建议
4. **及时响应** - 24 小时内响应审查请求
5. **认可好代码** - 不要吝啬赞美

### 作者

1. **保持小 PR** - 每次审查 < 300 行为宜
2. **清晰的描述** - 说明变更原因和影响范围
3. **自测充分** - 提交前确保功能正常
4. **积极回应** - 及时回复审查意见
5. **不急于合并** - 确保所有问题都妥善处理
